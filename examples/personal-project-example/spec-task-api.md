# Agent-Ready Specification: Personal Task Management API

## 1. Context (Business Problem)

### Current State
I currently track personal tasks in a spreadsheet. It's cumbersome:
- Takes 2 minutes to add a task
- Hard to query (e.g., "what's due this week?")
- No API to integrate with other tools
- Manual backup required

### Desired State
A simple REST API I can call from anywhere (CLI, web, mobile apps) that:
- Stores tasks with metadata
- Supports queries and filtering
- Is fast and reliable
- Runs locally

### Business Value

**Primary KPI Impact:**
- Metric: Time to add a task
- Current: 2 minutes (spreadsheet)
- Target: 10 seconds (API call)
- Measurement: Manual timing

**Secondary Benefits:**
- Learn FastAPI for work projects
- Practice agentic development workflow
- Create reusable component for other projects

**Risk if Not Done:**
- Continue with inefficient manual process
- Miss opportunity to learn new framework

## 5. Edge Cases & Error Handling

### Edge Cases

| Scenario | Expected Behavior | Test Coverage |
|----------|------------------|---------------|
| Task with very long title (>255 chars) | Truncate or reject with 422 | T009 |
| Due date in the past | Accept but warn (not block) | T010 |
| Concurrent task creation | Both tasks created with unique IDs | T011 |
| Database locked (SQLite) | Retry with exponential backoff | T012 |
| Invalid task ID format | 404 Not Found | T013 |
| Update non-existent task | 404 Not Found | T014 |
| Delete already deleted task | 404 Not Found | T015 |
| Search with special characters | Escape properly, return results | T016 |
| Empty task list | Return empty array, 200 OK | T017 |

### Error Handling

| Error Condition | HTTP Status | Error Code | User Message | Log Level |
|----------------|-------------|------------|--------------|-----------|
| Validation error | 422 | VALIDATION_ERROR | "Invalid input: [details]" | INFO |
| Task not found | 404 | NOT_FOUND | "Task not found" | INFO |
| Not authenticated | 401 | UNAUTHORIZED | "Authentication required" | WARN |
| Invalid token | 401 | INVALID_TOKEN | "Invalid authentication token" | WARN |
| Database error | 500 | DATABASE_ERROR | "Internal server error" | ERROR |
| Unexpected error | 500 | INTERNAL_ERROR | "Internal server error" | ERROR |

### Recovery Procedures

**Automatic Recovery:**
- Database connection retry: 3 attempts with exponential backoff
- JWT token refresh: Automatic on expiry

**Manual Intervention:**
- Database corruption: Restore from backup file
- Application crash: Restart uvicorn

## 7. Data Classification & Handling

### Data Types Involved

| Data Type | Classification | Source | Destination | Handling |
|-----------|---------------|--------|-------------|----------|
| Task content | Internal | User input | SQLite | Plain text |
| JWT token | Internal | Generated | Client | Signed, short-lived |
| Password hash | Internal | Config | Memory | Hashed (bcrypt) |

### Data Flow Diagram

```
User Input → FastAPI → Pydantic Validation → SQLAlchemy → SQLite
                ↓              ↓                    ↓
           JWT Check    Type Conversion       Auto-commit
```

### Retention & Deletion

**Retention Policy:**
- Task data: Indefinite (until deleted by user)
- JWT tokens: 24 hours
- Logs (if implemented): 30 days

**Deletion Procedure:**
1. DELETE /tasks/{id} removes from database

### API Design

**Base URL:** http://localhost:8000

**Endpoints:**
- POST /auth/login - Get JWT token
- GET /tasks - List tasks (with query params)
- POST /tasks - Create task
- GET /tasks/{id} - Get task
- PUT /tasks/{id} - Update task
- PATCH /tasks/{id} - Partial update
- DELETE /tasks/{id} - Delete task

**Query Parameters (GET /tasks):**
- priority: low, medium, high
- status: pending, completed
- due_before: ISO date
- due_after: ISO date
- search: text search
- sort: field name
- order: asc, desc
- skip: pagination offset
- limit: page size (max 100)
